Critical learnings baked into the code (don't relearn the hard way):
- **HVAR gotcha**: Recursive ships HVAR+gvar; HarfBuzz reads advances from HVAR.
  New glyphs absent from HVAR's AdvWidthMap repeat the last entry's +700 wght
  delta → ballooned to 800 at heavy weights. `repair_hvar` pins new glyphs to a
  zero-delta varidx. HVAR's map decompiles lazily → force-decompile
  `font["HVAR"].table.AdvWidthMap.mapping` at build start (done). fontTools'
  instancer HIDES this bug (maps HVAR by name) — only HarfBuzz exposes it.
- **Lookup ordering**: long-arrow lookups are built BEFORE lilx lookups (lower
  indices) so HarfBuzz applies them first; otherwise lilx's connected-dash sub
  eats the dashes and `--->`/`<--->` don't form. Keep this order.
- Recursive VF has **no `calt`**; code ligatures live in `dlig`. dlig ligates
  only `--`/`---`/bounded arrows; 4+ dash runs stay loose.
- HarfBuzz can't propagate right→left, so right arrows are bounded (MAX_RUN=14).
- Heavy weights: Lilex caps at wght 700, so curvy parens etc. render lighter than
  native Recursive Black (accepted; can't per-weight fall back in one VF glyph).

## Open Questions / Risks
- **Coexistence**: the static build (8 instances) and the VF both use family
  "Recursive KG". If both are installed, apps may merge/pick unpredictably —
  recommend using one (the VF covers all weights via axes). Distinct PS names
  mitigate the worst clash.
- **Gated arrows tradeoff**: the 12 arrows are gated behind `lilx` via .notdef
  placeholders, so cmapping them suppresses OS font-fallback for those codepoints
  when lilx is off. Flip to always-on is a one-line change (cmap straight to the
  arrow). Documented in the Decision Log.
- **ss13** references existing lookup indices `[82,85,87,89,90]`; if the source
  Recursive VF version changes, re-derive via `feature_lookup_indices`.
- Static-build acceptance checkboxes in the plan are unticked but the static build
  was completed/installed earlier (commit `5388a48`); not VF-relevant.
